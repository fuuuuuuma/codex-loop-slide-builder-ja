# make-slides — Codex × ループでスライドを半自動生成

「AIにゴールと合格条件だけ渡して、生成 → 検証 → 改善 を自分で回させる（ループエンジニアリング）」を、
スライド作成にそのまま適用する道具一式です。**実際に全140枚のセミナースライドを1セッションで仕上げた往復の型**を、誰でも再現できる形に落としています。

---

## ⭐ このリポジトリを渡されたAIへ（最初に必ず読む）

あなた（Codex / Claude Code などのAIエージェント）が、**このリポジトリのURLやフォルダを渡されたら**、
まず内部を読み込み、相手の状況に合わせて次の3モードのどれかで動いてください。**いきなりスライドを作り始めない。**

### 手順1：まず内部を読む
- `README.md`（資料全体）／ `kit/README.md`（キットの使い方）
- `kit/design-systems/INDEX.md`（DESIGN.md 50種の一覧）
- このファイル `kit/make-slides/SKILL.md` ＋ `AGENTS.md`（守るルール）＋ `prompts.md`（Phase別プロンプト）

### 手順2：モードを判定する
| 相手の状況 | モード |
|---|---|
| 作りたいスライドのテーマ・構成・伝えたいことが**まだ固まっていない** | **A 壁打ち** |
| 作るものは決まっていて、**土台（フォルダ一式）がほしい** | **B テンプレート作成** |
| 材料（台本・記事・要点）が**揃っている** | **C 作成** |

迷ったら **A（壁打ち）から**。状況が変わったら B / C に移る。

### モードA：壁打ち（質問で引き出す）
**一度に全部聞かない。1〜2問ずつ**会話で引き出す。

1. 誰に向けたスライド？（聞き手・場面）
2. 一番伝えたいことを1行で？（コアメッセージ）
3. 何枚くらい／どこで使う？（尺・用途）
4. 雰囲気は？ → `kit/design-systems/INDEX.md` から**2〜3個に絞って提案**し、選んでもらう
5. 手元の材料は？（台本／記事／箇条書き／まだ無い）

引き出せたら、相手の意向で **B（土台を作る）** か **C（そのまま作る）** に進む。

### モードB：フォルダテンプレート作成（土台をscaffold）
相手の決定をもとに、作業フォルダを1つ作る。**選ばれた DESIGN.md とテンプレ群をコピーするだけ**：

```
<相手のプロジェクト名>/
├── AGENTS.md     ← kit/make-slides/AGENTS.md をコピー（Codex の運用ルール）
├── DESIGN.md     ← kit/design-systems/NN-….DESIGN.md（選んだテーマ）をコピー
├── outline.md    ← kit/make-slides/outline.template.md をコピー
├── script.json   ← kit/make-slides/script.example.json をコピー（中身は空に）
├── slides.html   ← kit/make-slides/slides.template.html をコピー
└── check.mjs     ← kit/make-slides/check.mjs をそのままコピー
```

`slides.html` の `:root{…}` を、選んだ DESIGN.md の `:root` ブロックで置き換える。
作り終えたら「土台ができました。次は Phase 1（章立て）からいきましょう」と促す。

### モードC：作成（ループを回す）
`prompts.md` の **Phase 1 → 4** を順に回して、スライドを完成させる（下記「手順」）。

---

## このスキルが作るもの

- **単一HTMLのスライド**（1280×720・ブラウザで開くだけ・配布も簡単）
- **1スライド1メッセージ**・**はみ出しゼロ**・**通し番号は自動**

## 同梱ファイル（役割分担＝AIが迷わない土台）

| ファイル | 役割 |
|---|---|
| `AGENTS.md` | Codex が毎回守る社内ルール（禁止ワード・1スライド1メッセージ・数値は出典必須） |
| `DESIGN.md` | 配色・フォント・型の正典。`../design-systems/` から1つ選んでコピー |
| `outline.template.md` | 構成（章立て・見出し）のひな型 |
| `script.example.json` | 1スライド = 1ブロックのデータ例（**文章と見た目を分ける**） |
| `slides.template.html` | 空のスライド骨格。**番号・カウンターはJSが自動採番** |
| `check.mjs` | はみ出し検出（評価役＝ループの心臓） |
| `prompts.md` | Phase別のコピペ用プロンプト |

## 手順（モードCで回すループ）

```
Phase 0 準備 … 伝えたい結論（コアメッセージ）を1行で決める          ← 人（監督）
Phase 1 骨子 … outline.md に章立て（見出しだけ）を作る              ← AI＋人
Phase 2 生成 … script.json に1枚ずつ本文（1スライド1メッセージ）    ← AI
              → slides.html に流し込む
Phase 3 改善 … 批評 → 改善を 3〜5周。死にスライド排除・ヘッダー統一  ← AI（ループ）
Phase 4 検証 … check.mjs で はみ出し0 まで直す。枚数・番号を整合     ← AI（評価役）
```

各Phaseのコピペ用プロンプトは `prompts.md` にあります。

## 守ること（`AGENTS.md` と共通）

1. **1スライド1メッセージ**。言いたいことを詰め込まない。
2. **文章（script.json）と見た目（slides.html）を分ける**。中身だけ直しても見た目が崩れない。
3. **数値・価格・日付は出典が取れなければ書かない**。確度マーク（✅公式 / 🔶一次情報 / ⚠️未確認）を各主張に付ける。
4. **誇張・断定保証をしない**（必ず／絶対／確実／神／ヤバい／裏技 等は使わない）。
5. **通し番号・枚数はハードコードしない**。スライドの数を数えて流し込む（`slides.template.html` のJSが自動）。
6. **死にスライドは「説明が弱い方」を消す**（同じ主張の重複は、専門的すぎる版を消して分かりやすい版を残す）。
7. **参加・思考をうながす問いを 5〜8枚に1回**入れて、聞く側のリズムを作る。

## ループの肝（なぜ回るのか）

スライドは「良し悪しを**出力で○×が付けられる**」タスクです。だから「**作る役（生成）と採点する役（check.mjs）を分け、両方OKだけ合格**」にすれば、AIが自分で「はみ出してる→直す」を回せます。

- 文字がはみ出していないか → `check.mjs` が高さを測って判定
- 1スライド1メッセージか → 見出し・本文量で判定
- 枚数・通し番号が合っているか → 数えれば分かる
- 死にスライドが無いか → 重複を検出

## Codex で使うときのメモ（このチャンネルの推し）

- ルールは **`AGENTS.md`**（Codex が読む運用ルール）。階層的に置けて、現在地に近いファイルほど優先。
- 役割を分けたいときは **`.codex/agents/*.toml`**（1ファイル＝1担当：構成担当／文章担当／検証担当）。
- ゴール条件まで自走させたいときは **Codex の `/goal`**（はみ出し0・全◯枚 を渡す）。
- このスキル自体は `SKILL.md`。Codex の Skills として置くか、フォルダごと渡す。

## 発展（凝るときだけ）

- **コメントアンカー編集**：スライドは `<!-- 番号 名前 -->` ＋ `<section>` の単位。番号でなく**コメント文字列をアンカー**に「削除／差し替え／直前挿入」すると、番号がズレても安全。
- **画像**：外部URL参照は配布時にパス切れ。`curl` → 縮小 → **base64 で `data:` URIに埋め込み**、単一HTML自己完結を保つ。
- **役割分担**：見た目はデザイン特化ツール、構造（文章・骨組み）はCodex、と分けると凝っても崩れにくい。
- **PPTX化**：完成スライドを2倍スクショ → `python-pptx` で 13.333×7.5inch に画像貼り（崩れゼロ）。

> このスキルは「撮影前の手元デッキを量産する」用途のテンプレートです。営業資料・解説スライド・動画用デッキにそのまま展開できます。
