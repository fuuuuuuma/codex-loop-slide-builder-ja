## 1. この資料は何か

これは「Codex（OpenAIのコーディングエージェント）に “ループ” を組ませて、**スライド作成を半自動で量産する**」やり方を、撮影前に手元へ置くためのリサーチノートです。土台にしているのは、実際に **AIエージェント／AI収益化ラボのセミナースライド（全140枚）** を1セッションで「フィードバック → 修正 → 検証 → 公開」と往復して仕上げた制作ログ。その往復の“型”を、誰でも再現できる形に一般化しています。

受け取った人が、この資料とセットで手にできるのは次の3つです。

1. **スライド作成のファイル構造テンプレート** — 「どのファイルに何を書くか」をAIと共有するための役割分担セット。AIが迷わず作業できる土台になります。
2. **mdファイルを読み取って「Skill（手順の型）」にして配る方法** — 一度うまくいった作り方を `SKILL.md` という1ファイルにまとめ、`/コマンド` 一発で呼び出せる “あなた専用の手順” にして、チームやお客さんに配れます。
3. **スライドテンプレート集** — 自前の「DESIGN.md 50選」（配色・レイアウトの引き出し）と、外部に公開されている優良テンプレ集へのリンク。AIに「これを参考に」と渡すだけで、デザインの当たり外れが減ります。

この3つは、後で説明する **ループエンジニアリングの4つの層**（プロンプト → 参照情報 → 手順 → 反復）にそのまま対応しています。バラバラの便利グッズではなく、「ループでスライドを回す」ための一式です。

## 2. ループエンジニアリングとは

むずかしく考えなくて大丈夫です。ひとことで言うと——

> **「新人さんに “ゴール” と “合格ライン” だけ伝えて、あとは自分で直しながら仕上げてもらう」** ——これをAIにやらせるのがループエンジニアリングです。

ふつうのAIの使い方は「1回お願いして、出てきたものを自分で直す」の繰り返し。これだと、毎回こちらが指示を考え直す必要があります。ループエンジニアリングは発想が逆で、**「やり直しの判断」までAIに渡してしまう**。具体的には、AIに次の流れを“自分で”回させます。

```
① 作る  →  ② 自分でチェックする  →  ③ ダメなら直す  →  ①に戻る  …（合格するまで）
```

ポイントは2つです。

- **ゴール（何ができたら完成か）を、言葉ではなく “確かめられる形” で渡す**。たとえば「文字がはみ出していない」「1枚に言いたいことが1つだけ」「全部で◯枚」のように、機械が○×を判定できる条件にします。
- **作る役（ジェネレーター）と、チェックする役（エバリュエーター）を分ける**。同じAIに「作って、自分で採点して」と言うと甘くなりがちなので、チェックは別の小さな仕組み（後述の自動チェックや別エージェント）に任せます。これは画像生成AIの「描く側／見抜く側」を競わせる考え方（GAN）と同じ発想です。

この「ゴールだけ渡して反復で仕上げさせる」型を、エンジニアたちはこう言っています。

- Addy Osmani（Google・著名エンジニア）:「ループは仕事を“変える”のであって、あなたを仕事から消すわけではない。**あなたは “それをやる仕組み” を設計する側になる**」
- Boris Cherny（Claude Code 作者）:「もうAIにプロンプトを書かない。**ループを書くのが仕事**になった」
- Peter Steinberger:「もうエージェントに毎回プロンプトするな。**ループを設計しろ**」

> 🔶 上記の引用は各氏の発信（記事・配信・X）に基づく要約です。原文ニュアンスは出典先で確認してください。

## 3. なぜ「スライド作成」はループ化に最適なのか

ループエンジニアリングが効くのは「**ゴールを “出力で確かめられる” タスク**」です。コードが分かりやすい例で、「`テストが通る（exit 0）`」「`git status がクリーン`」のように、合格・不合格を機械が判定できます。

スライド作成も、実は同じ性質を持っています。「良いスライド」を、次のように **機械が○×を付けられる条件** に落とせるからです。

- **文字がはみ出していないか**（要素の高さが枠を超えていないか）→ ブラウザで自動測定できる
- **1スライドに言いたいことが1つだけか**（1スライド1メッセージ）→ 見出しの数・本文量でチェックできる
- **全体の枚数・通し番号が合っているか** → ファイル内の `スライドの数` を数えれば自動で確かめられる
- **死にスライド（無くても困らない・重複している）が無いか** → 同じ主張の重複を検出できる

つまりスライドは「見た目の作業」に見えて、**合否を数えられる作業**。だからループにすると、AIが「はみ出してる → 直す」「重複してる → 消す」を自分で回せます。実際の制作ログでも、反復の“型”はこうなっていました。

```
編集 → はみ出し検証（自動）→ スクショ目視 → 体裁を再生成 → 公開 → 報告
```

この1周を、毎回こちらが手で回すのではなく、**Codexに「合格条件」を渡して自動で回させる**——それがこの資料のゴールです。

## 4. 全体像：Codex × ループでスライドができるまで

X で発信されている実践手法（@uchita_success ほか）と、上の制作ログを合わせると、流れはおおむね **Phase 0〜4** に整理できます。

| Phase | やること | 誰が主役 |
|---|---|---|
| **Phase 0：準備** | テーマ・ターゲット・伝えたい結論（コアメッセージ）を1行で決める | 人（監督） |
| **Phase 1：骨子** | 全体の構成（章立て・各スライドの見出し）をまず作る | Codex＋人 |
| **Phase 2：1枚生成** | 1スライドずつ、**1スライド1メッセージ**で本文を生成 | Codex |
| **Phase 3：改善ループ** | 1枚ごとに「批評 → 改善」を **3〜5周** 回す（死にスライド排除・ヘッダー統一） | Codex（ループ） |
| **Phase 4：検証** | はみ出し・枚数・体裁を自動チェックし、ダメなら直す | Codex（評価役） |

> 🔶 Phase の分け方は X で共有されている手法を整理したもので、固定の公式手順ではありません。要は「いきなり全部作らせない。**構成 → 1枚ずつ → 反復改善 → 自動検証** の順に分ける」のが肝です。

ここで大事な “役割分担” の発想があります。@uchita_success の発信では、**ビジュアル（見た目）を整える役と、構造（骨組み・文章）を磨く役を分ける**——たとえば見た目はデザイン特化のツールに、構造はCodexに、と分担すると、両方が中途半端にならずに済む、という上級テクが紹介されています。最初は分けなくてもOKですが、「凝りだすと崩れる」と感じたら役割を分けてください。

## 6. 柱①：スライド作成のファイル構造テンプレート

AIに「いい感じにスライド作って」と丸投げすると、毎回バラバラのものが出てきます。理由は **AIが “どこに何を書くか” を知らない** から。そこで、最初に「ファイルの役割分担」を決めて渡します。これがあるだけで、AIの仕事が安定します。

おすすめの最小セットはこれです（フォルダ1つに入れるだけ）。

```
slide-project/
├── AGENTS.md        ← AIへの“社内ルール”（口調・禁止事項・守る型）
├── DESIGN.md        ← 配色・フォント・余白・スライドの型（デザインの正典）
├── outline.md       ← 構成（章立て・各スライドの見出し）
├── script.json      ← 各スライドの中身（タイトル・本文・図の種類）を1枚ずつ
├── slides.html      ← 完成スライド本体（単一HTML・ブラウザで開ける）
└── check.mjs        ← 自動チェック（はみ出し検出・スクショ）＝“評価役”
```

それぞれの役割は次の通りです。

- **AGENTS.md** — Codex版の “社内ルールブック”。「禁止ワード」「1スライド1メッセージを守る」「数値は出典が無ければ書かない」などを書いておくと、AIが毎回それに従います。
- **DESIGN.md** — デザインの正典。色コード・フォント・余白・「表紙はこう／本文はこう」の型を書きます。これが柱③の「50選」とつながります。
- **outline.md** — 先に骨組みだけ作る場所。ここを固めてから本文に入ると、後半が崩れません。
- **script.json** — 1スライド = 1ブロックで中身を持つデータ。**「文章（中身）」と「見た目（HTML）」を分けておく**のがコツ。中身だけ直せば見た目は崩れません。
- **slides.html** — 実際に映すスライド。1280×720の単一HTMLにしておくと、ブラウザで開くだけで確認でき、配るのも簡単です。
- **check.mjs** — **ここがループの肝**。スライドを開いて「はみ出している場所」を機械的に見つけ、スクショを撮る小さな道具。これがあるから、AIは「自分でチェックして直す」が回せます。

> ✅ この「文章と見た目を分ける」「番号は数えて自動で振る」構造は、実際の140枚デッキでも使われた実証済みの形です。番号をハードコードせず **スライドの数を数えて流し込む** ので、ブロックを足し引きしても通し番号が自動でそろいます。

## 7. 柱②：mdファイルを読ませて「Skill」にして配る

一度うまくいった作り方を、毎回ゼロから説明し直すのは大変です。そこで、**手順を1つの `SKILL.md` というファイルにまとめて、`/コマンド` 一発で呼び出せる “型” にして配ります**。これが「mdを読み取ってSkill化する」の意味です。

### Skill とは何か（やさしく）

Skill ＝「**いつ使うか＋やることの手順**」を書いた取扱説明書。AIエージェント（Claude Code や Codex）は、この説明書を読み込むと、その手順どおりに動けるようになります。たとえば「スライドを作る手順」をSkillにしておけば、次回からは `/make-slides` と打つだけで、構成→生成→改善→検証まで一気に走ります。

### 作り方（3ステップ）

1. **元になるmdを用意する** — 今回の制作ログのように、「どういう順番で・何を・どう確かめたか」を書いた手順メモ（Markdown）を1本用意します。
2. **AIに「これをSkillにして」と頼む** — AIが内容を読み、`SKILL.md` の形（下記）に整えてくれます。
3. **フォルダに置いて配る** — `SKILL.md` を所定の場所に置くか、GitHubで公開すれば、受け取った人も同じ手順を `/コマンド` で使えます。

### SKILL.md の最小構造

```markdown
---
name: make-slides
description: 構成→1枚ずつ生成→批評改善→自動検証で、はみ出しゼロのスライドを作る。
             「スライド作って」「資料にして」で使う。
---

## 手順
1. outline.md に章立てを作る（1スライド1メッセージ）
2. script.json に1枚ずつ本文を書く（中身と見た目を分ける）
3. slides.html を生成する
4. check.mjs ではみ出しを検出 → あれば直す（3〜5周）
5. 通し番号・枚数を数えて整合させる
```

ポイントは **`description`（いつ使うか）を具体的に書く** こと。ここが曖昧だと、AIが「今これを使うべきか」を判断できません。「スライド」「資料」「プレゼン」など、実際に言いそうな言葉を入れておきます。

> 🔶 Skillの仕組み（説明書を読み込んで手順どおり動く）は Claude Code / Codex 共通の考え方です。配置場所やファイル名の細部は各ツールの最新ドキュメントで確認してください。

## 8. 柱③：スライドテンプレート集

デザインの当たり外れを減らす一番かんたんな方法は、**「良いお手本」をAIに渡す**ことです。柱③は、その “お手本の引き出し” を2種類用意します。

### (A) 自前の「DESIGN.md 50選」

`DESIGN.md` に、自分の “勝ちパターン” を貯めていきます。最初から50個そろえる必要はありません。1本作るたびに「この配色は良かった」「この型は読みやすい」を1行ずつ足していくと、自然と引き出しが増えます。たとえばこんな項目を貯めます。

- **配色パターン**：ネイビー×ブルー（クール）／コーラル×クリーム（親しみ）／エメラルド×ゴールド（お金・信頼）…
- **スライドの型**：表紙／結論3行（TL;DR）／3本柱カード／比較表／before-after／フロー図／サイクル図／FAQ／まとめ…
- **守るルール**：濃い差し色は “文字” に使わない（薄くて読めない）／表の見出しは濃い地に白文字／1スライド1メッセージ…

この `DESIGN.md` を柱①のフォルダに入れておけば、Codexは毎回それを参照して、ブレない見た目で作ってくれます。**「50選」は “完成品” ではなく “育てる資産”** だと考えてください。

### (B) 外部の公開テンプレート集（参考リンク）

ゼロから考えず、世の中の優良テンプレを “参考画像” として渡すと早いです。実在する代表的なものを挙げます（利用前に各サイトのライセンス＝商用可否・クレジット表記・再配布の可否を必ず確認してください）。

- **slide lib（スライドリブ）／ c-slide** — `https://cone-c-slide.com/lib/` 。株式会社Cone運営のBtoB資料ポータル。営業資料・会社紹介などのテンプレが豊富で、資料作成代行で培った “伝わる構成” の見本が多いのが特長。✅
- **SlidesCarnival** — `https://www.slidescarnival.com/` 。PowerPoint / Googleスライド向けの無料テーマ大規模ライブラリ。✅
- **Slidesgo** — `https://slidesgo.com/` 。ビジネス〜教育まで幅広いトレンドデザイン。✅
- **Slidesacademy** — `https://www.slidesacademy.com/jp/` 。無料テーマの大規模ライブラリ。✅
- **まとめ記事**：Workship MAGAZINE「Googleスライドのテンプレ配布サイト21選」`https://goworkship.com/magazine/googleslide-template-matome/` ✅

> ⚠️ テンプレは「丸ごと提出物に使う」のではなく、AIに渡す **“こういう雰囲気・構成で” の参考** として使うのが安全です。ライセンスは媒体ごとに違うので、ダウンロード前に必ず確認してください。

## 9. 使い方（初心者基準）

専門用語ゼロで、まず触ってみる手順です。

### こんな悩み → こう頼むだけ

| こんな悩み | Codex にこう頼む |
|---|---|
| 「構成から考えるのがしんどい」 | 「このテーマで、スライドの章立て（見出しだけ）を10枚分作って」 |
| 「1枚が文字だらけになる」 | 「各スライドを “言いたいこと1つだけ” に削って」 |
| 「文字がはみ出す・崩れる」 | 「はみ出しを自動でチェックして、直るまで直して」 |
| 「毎回同じ説明をするのが面倒」 | 「今の手順を SKILL.md にまとめて」 |

### すぐ試せるプロンプト（コピペ可）

```
あなたはスライド作成の担当です。次のルールで作ってください。
- 1スライド = 言いたいこと1つだけ（1スライド1メッセージ）
- 文字のはみ出しは禁止。最後にブラウザで高さを測って確認する
- まず outline.md に章立て → script.json に本文 → slides.html を生成 の順
- 出典の取れない数字・価格は書かない
テーマは「◯◯」。まずは章立て（見出しだけ10枚分）を出してください。
```

### Phase別プロンプト（ループを“組む”言い方）

```
# Phase 1 骨子
「◯◯のスライドの章立てを作って。各スライドは見出し1行＋一言メモだけ」

# Phase 2 1枚ずつ
「outline.md の3枚目を、本文スライドに起こして。1スライド1メッセージで」

# Phase 3 改善ループ（ここが“ループ”）
「各スライドを self-review して。死にスライド（無くても困らない/重複）を見つけて
 統合・削除。合格するまで批評→改善を最大5周くり返して」

# Phase 4 検証
「check.mjs を実行して、はみ出しが出たスライドを直して。
 はみ出しゼロになるまで自動でくり返して」
```

### 実践ステップ

1. フォルダを1つ作り、`AGENTS.md` / `DESIGN.md` / `outline.md` を置く（柱①）。
2. Codexに「このルールで章立てから作って」と頼む（Phase 1）。
3. 1枚ずつ本文を生成（Phase 2）。
4. 「批評 → 改善を5周」「はみ出しゼロまで直して」でループを回す（Phase 3〜4）。
5. うまくいったら、その手順を `SKILL.md` にして次回から `/コマンド` 一発に（柱②）。

## 10. 検証＝“評価役”の作り方

ループの心臓は **「機械が○×を付けられる検証」** です。ここが甘いと、AIは “それっぽいけど崩れている” もので満足してしまいます。スライドで効く検証は主にこの3つ。

1. **はみ出し検出（自動）** — ブラウザでスライドを開き、各要素の高さが枠を超えていないかを測ります。実際の制作では Playwright という道具で全スライドを描画し、はみ出しが無ければ「合格」、あれば該当スライドを直す、を自動化しました。
2. **枚数・通し番号の整合** — スライドを1枚足すと「全◯枚」の数字がズレます。これを **手で書かず、ファイル内のスライド数を数えて自動で流し込む** ことで、ページ番号・目次・「全◯枚」が常に一致します。
3. **死にスライドの排除** — 同じ主張が二重に載っていたら、**“説明が弱い方” を消す**のがコツ。専門的すぎる版を消し、業務例で書いた分かりやすい版を残します。

そして大原則は **「作る役」と「採点する役」を分ける**こと。同じCodexに採点させると甘くなるので、検証は別の小さな道具（check.mjs）や別エージェントに任せ、**両方が「OK」と言ったものだけ合格**にします。これがループの精度を決めます。

> ✅ 「自己言及する数値（このスライドは何枚？等）は手で書かず、実数を数えて流し込む」は、制作ログで実証された再発防止策です（枚数が変わるたびに答えが自動で追従）。

## 11. 比較表

| 観点 | ① 1枚ずつ手作り | ② AIに丸投げ1発 | ③ Codex × ループ |
|---|---|---|---|
| 速さ | 遅い | 速いが当たり外れ大 | **速い＋安定** |
| 体裁（はみ出し等） | 自分で全部確認 | 崩れても気づかない | **自動で検出・修正** |
| 1スライド1メッセージ | 守れるが手間 | バラつく | **ルールとして強制** |
| 作り直しの判断 | 自分 | 自分（毎回指示） | **AIが自分で回す** |
| 量産・再現性 | 低い | 中（毎回ゼロから） | **高い（Skillで型化）** |
| 向き不向き | 1〜2枚の単発 | ラフな下書き | **本数を量産・配布する人** |

「③が常に正解」ではありません。**1〜2枚をサッと作るなら①や②で十分**。ループが効くのは「**毎週・毎本つくる**」「**チームで同じ品質をそろえたい**」「**配って横展開したい**」ときです。

## 12. もう一歩進んだ使い方

ここは本筋ではないので、慣れてきたら、で十分です。

- **AGENTS.md で “社内ルール” を効かせる** — Codexは、フォルダに置いた `AGENTS.md`（運用ルール）を読み込みます。階層的に置けて、現在地に近いファイルほど優先されます。「禁止ワード」「口調」「数値は出典必須」などを書いておくと、全工程でブレません。
- **サブエージェントで役割分担** — Codexは `.codex/agents/` に “担当別AI”（TOMLファイル1つ＝1担当）を置けます。たとえば「構成担当」「文章担当」「検証担当」を分けると、1人のAIに全部やらせるより安定します。
- **ビジュアルと構造を分担** — 前述のとおり、**見た目はデザイン特化ツール、構造（文章・骨組み）はCodex**、と分けると凝ったデザインでも崩れにくくなります（@uchita_success の上級テク）。

> 🔶 Codex の AGENTS.md / サブエージェントの細かい仕様（既定値・配置）は時点により変わります。最新は OpenAI の Codex ドキュメントで確認してください。

## 13. FAQ

**Q. プログラミングができなくても使えますか？**
A. 章立て・本文づくり・体裁チェックは、日本語で頼むだけで進みます。最初は「章立てして」「1枚1メッセージに削って」「はみ出し直して」の3つだけで十分です。

**Q. ループって、ずっと回り続けて止まらないのでは？**
A. 「合格条件（はみ出しゼロ・全◯枚など）を満たしたら止まる」と決めておきます。条件を “出力で確かめられる形” にしておくのが、安全に止めるコツです。

**Q. PowerPointじゃなくてHTMLなの？**
A. 単一HTMLにすると、ブラウザで開くだけで確認でき、はみ出しの自動チェックや配布が簡単です。必要なら、完成したスライドを画像にして PowerPoint へ貼る形にもできます。

**Q. 本当に時短になりますか？**
A. 1〜2枚なら手作りでも変わりません。効果が出るのは「量産・横展開・チーム統一」のとき。効果は使い方しだいで、必ず◯%短縮、とは言えません。
