ループエンジニアリングとは

ひとことで言うと 「新人にゴールと合格ラインだけ伝えて、自分で直しながら仕上げてもらう」 をAIにやらせること。ふつうは「1回頼んで自分で直す」の繰り返しですが、ループは “やり直しの判断” までAIに渡します。

ループの「1周」
① 作る
1スライド生成
→
② チェック
はみ出し・重複
→
③ 直す
ダメな所を改善
↻
④ 合格判定
条件を満たすまで

※ 肝は 「作る役」と「採点する役」を分けること。自分で採点すると甘くなる。🔶「ループエンジニアリング」は2026年に広まった呼び名で公式用語ではない

なぜ“スライド”がループ最適か

ループが効くのは「ゴールを 出力で○×が付けられる」タスク。スライドはまさにそれ。

  • 文字が はみ出していない か → ブラウザで高さを測れる
  • 言いたいことが 1つだけ か → 見出し数で測れる
  • 枚数・通し番号 が合っているか → 数えれば分かる
  • 死にスライド（重複・不要）が無いか → 検出できる
流れは Phase 0〜4 に分ける
Phaseやること主役
0 準備コアメッセージを1行で決める人（監督）
1 骨子構成（見出しだけ）を作るCodex＋人
2 生成1スライドずつ本文（1メッセージ）Codex
3 改善批評→改善を3〜5周・死にスライド排除Codex（ループ）
4 検証はみ出し・枚数・体裁を自動チェックCodex（評価役）

🔶 Phase分けはX発信の手法を整理したもの。「いきなり全部作らせない」のが肝

柱① ファイル構造テンプレート

「いい感じに作って」と丸投げすると毎回バラバラになるのは、AIが“どこに何を書くか”を知らないから。先に役割分担を渡します。

# フォルダ1つに、この6ファイルを置くだけ
slide-project/
├─ AGENTS.md # AIへの社内ルール（禁止事項・守る型）
├─ DESIGN.md # 配色・フォント・型の正典
├─ outline.md # 構成（章立て・見出し）
├─ script.json # 1枚ずつの中身（文章と見た目を分ける）
├─ slides.html # 完成スライド本体（単一HTML）
└─ check.mjs # はみ出し検出＝評価役
各ファイルの役割
ファイル役割（ひとことで）
AGENTS.mdAIが毎回守る社内ルール（禁止ワード・1スライド1メッセージ等）
DESIGN.md色・フォント・型の正典。柱③「50選」とつながる
outline.md先に骨組みだけ作る場所。後半が崩れない
script.json文章と見た目を分ける。中身だけ直せる
check.mjsループの心臓。はみ出しを自動で見つける

使い方（初心者基準）

こんな悩み → こう頼むだけ
  • 「構成からがしんどい」→「章立て（見出しだけ）を10枚分作って」
  • 「1枚が文字だらけ」→「言いたいこと1つに削って」
  • 「はみ出す・崩れる」→「はみ出しを自動チェックして直して」
  • 「毎回同じ説明が面倒」→「今の手順を SKILL.md にまとめて」

FAQ・もう一歩

プログラミングができなくても使える？
章立て・本文・体裁チェックは日本語で頼むだけ。まず「章立て」「1枚1メッセージに削る」「はみ出し直す」の3つでOK。
ループって止まらないのでは？
「はみ出し0・全◯枚」など合格条件を満たしたら止まると決めておく。条件を“確かめられる形”にするのがコツ。
本当に時短になる？
1〜2枚なら変わらない。効くのは量産・横展開・チーム統一のとき。効果は使い方しだい（断定しない）。
もう一歩進んだ使い方
  • AGENTS.mdで社内ルールを効かせる（禁止ワード・口調・数値は出典必須）
  • サブエージェント（.codex/agents/）で「構成／文章／検証」担当を分ける
  • ビジュアルはデザイン特化ツール、構造はCodex、と担当を分ける（🔶@uchita_success）

🔶 Codexの細かい仕様（既定値・配置）は時点で変わる。最新は公式ドキュメントで確認。⚠️ 数値（◯時間→◯分等）は実績に差し替え・出典の取れない数字は書かない
